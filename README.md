# Claude Team

A multi-agent "company" for software product development. Drop into any directory as `<your-dir>/.claude/` — it acts as Company HQ (strategy, scoping, tech direction, executive reporting) for any product project beneath it.

> **Path note:** every path in this repo is relative to wherever YOU put the `.claude/` directory. Internal docs use `<tier1-hq>/.claude/` or `<workspace>/.claude/` as generic placeholders. Run it from any directory that makes sense for your work — at the user level, at a code-projects parent, or scoped to a single product line.

**The user is the CEO.** The team gathers requirements, drafts PRDs, picks tech stacks, generates per-project execution teams, and reports back. The user only weighs in at meaningful checkpoints.

## How It Works

```
You (CEO)  ⇄  Intake  →  Strategist  →  Architect  →  Tier 2 (per-project team)
              + Critic     + Critic       + Critic        ↓
              + EA         + EA           + EA          MVP shipped
                                           ↓               ↓
                                      Conductor        Reportback to HQ
                                           ↓
                                     State Machine
                                           ↓
                                     Org Designer (watches, proposes splits)
```

You talk to **Intake** when you have new work, scope shifts, or want to interview yourself into clarity.
You hear from **Executive Assistant (EA)** when there's something to decide, summarize, or know.
Everything else happens behind the scenes.

## The Founding Team (7 Agents)

| Agent | Role | Talks to |
|---|---|---|
| **Intake** | Director of Product Discovery — interviews you, briefs the team | You ⇄ Team |
| **Executive Assistant** | Chief of Staff — surfaces decisions, delivers briefings | Team → You |
| **Conductor** | CTO/CPO — routes work, enforces state machine, runs consistency checks | Team-internal |
| **Strategist** | VP Product — turns briefs into PRDs | Conductor-driven |
| **Architect** | VP Engineering — turns PRDs into scope + tech strategy + Tier 2 scaffold | Conductor-driven |
| **Critic** | Independent advisor — adversarial review of every artifact | Reads everything |
| **Org Designer** | Head of People — watches team performance, proposes splits & new roles | Slow cadence |

The team grows over time. Future agents (GTM Strategist, Growth Analyst, Customer Researcher, etc.) live as stubs in `agents/_planned/` until real friction triggers their activation.

## How to Start

**You have new work?**

Just talk. Tell Claude what you want to build. Intake will engage automatically and ask hard-hitting questions until it has a brief downstream agents can act on.

Or invoke directly:
```
/team "I want to build a music discovery site"
/intake
```

**You want to know what's happening?**

```
/status
/briefing
/queue        # decisions waiting on you
/inbox        # flagged items + FYI
```

**You think the team needs to evolve?**

```
/grow-team    # invokes Org Designer for a team-shape review
```

## The State Machine

Every project moves through 12 phases:

```
seed → intaking → briefed → stratego → prd-ok → scoping →
planned → scaffold → handed-off → shipped → measured → retro
```

Plus side-states: `paused`, `pivoted`, `abandoned`.

**5 hard checkpoints** stop for your approval (briefed, prd-ok, planned, scaffold, shipped). Everything else auto-advances. EA notifies you when something needs attention.

See `protocols/state-machine.md` for full contract.

## Tier 1 vs Tier 2

- **Tier 1 (this `.claude/`):** Company HQ. Framework-agnostic. Strategy, planning, scaffolding.
- **Tier 2 (each project's own `.claude/`):** Product team. Generated by Architect. Framework-specific (e.g., Next.js agents for a Next.js project, Swift agents for an iOS app).

Tier 2 receives a charter (handoff package). Tier 2 reports up via reportback channel. Tier 1 owns the "what and why," Tier 2 owns the "how."

## Public / Private Split

This repo is open-source-ready by design.

**Public** (ships in the repo): agent prompts, protocols, templates, playbooks, examples, narrative changelogs.

**Private** (`.gitignored`, never published): your real `memory/` (product taste, stack preferences, lessons), `workspace/` (every project you've run), private playbooks.

Anyone forking this repo gets a working scaffold from `_examples/` directories.

## Memory (the team's brain)

Cross-project knowledge that compounds:

- `memory/product-principles.md` — what "good" means to you
- `memory/stack-preferences.md` — default stacks per project type
- `memory/audience-knowledge.md` — recurring ICPs
- `memory/patterns.md` — recurring decisions
- `memory/lessons-learned.md` — post-mortems with lessons
- `memory/agent-changelog.md` — narrative log of team evolution

Hygiene rule: every memory entry needs provenance (which project, what date) so it can be audited for freshness and shareability.

## Versioning

This whole `.claude/` directory is intended to be a git repo:
```
cd <your-dir>/.claude
git init
gh repo create <owner>/<repo-name> --private --source=. --push
```

Every meaningful change (agent edit, role split, prompt tune) gets its own commit.
