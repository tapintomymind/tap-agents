**Status:** Copy-source reference. This is **paste-able wiring**, NOT an auto-applied scaffolder. ## WHY — the failure this replaces

The obvious default, `npx husky init`, writes `prepare: husky` to `package.json`, which sets git's `core.hooksPath = .husky/_`. That `_` shim directory is **generated by husky and git-IGNORED** (`.husky/_/.gitignore` contains `*`). So it is **ABSENT in a freshly-created `git worktree`** that never ran `npm install`.

`core.hooksPath` lives in the **shared** `.git/config` and is stored as a **relative** path, so every worktree resolves it against its OWN top level — `<worktree>/.husky/_` — which doesn't exist without an install. Result: git finds no hooks directory and runs **NO hook at all, silently** — no error, no `--no-verify` needed. The primary local secret-scan guard (gitleaks) is therefore **inactive in ~all worktrees, which is where most implementation actually happens** (a project can have ~9 live worktrees at a time).

**The fix (two parts):**

1. Point `core.hooksPath` at **`.husky` itself** — a git-TRACKED directory (`.husky/pre-commit` is committed) that exists in every checkout WITHOUT an install. Done by a tiny `prepare` script (`scripts/setup-hooks.mjs`), explicitly **NOT** `prepare: husky`.
2. Make `.husky/pre-commit` **self-contained** — it does not source husky's `.husky/_/h` runtime (absent in fresh worktrees) and runs **gitleaks first and unconditionally** as a system binary, so the secret scan is never gated behind `npm install` having run.

Even with this fix, the pre-commit hook stays **best-effort** (bypassable via `--no-verify`, absent if gitleaks isn't installed). The **CI gate (`.github/workflows/gitleaks.yml`) is the authoritative checkpoint** — see §4 of the protocol for the authoritative-ordering posture. The CI workflow below triggers on `pull_request` to **both** `dev` and `main` precisely so a PR into the integration branch (`dev`) is scanned *before* merge, not only after.

## The four files

Copy all four. Replace `<project>` in the comments with the target project's name. Assumes a `.gitleaks.toml` config exists at repo root (gitleaks defaults are usually sufficient; the file can be empty-with-allowlist).

### 1. `package.json` — the `prepare` line

The lifecycle hook must be **`node scripts/setup-hooks.mjs`**, explicitly **NOT `"husky"`** (husky's own init is what sets the broken `.husky/_` hooksPath):

```json
{
  "scripts": {
    "prepare": "node scripts/setup-hooks.mjs"
  }
}
```

### 2. `scripts/setup-hooks.mjs`

The `prepare` script that pins `core.hooksPath=.husky`. Hardened version (PR #95 commit `b54c1b8`): try/catch around the `git config` exec so a config failure never fails `npm install`; `.git`-absent no-op; `HUSKY=0` opt-out; `pre-commit`-presence guard that fails loudly rather than configuring an empty hooks path.

```javascript
#!/usr/bin/env node
/**
 * scripts/setup-hooks.mjs
 *
 * `prepare` lifecycle script (runs on `npm install`). Pins git's
 * `core.hooksPath` to the git-TRACKED `.husky` directory so the pre-commit
 * hook fires in EVERY checkout — including fresh `git worktree` checkouts
 * that never ran `npm install`.
 *
 * ---------------------------------------------------------------------------
 * WHY THIS EXISTS  (fix/secret-scan-worktree-coverage)
 *
 * husky v9's own init (`prepare: husky`) sets `core.hooksPath = .husky/_`.
 * That `_` shim dir is generated by husky and git-IGNORED
 * (`.husky/_/.gitignore` = `*`), so it is ABSENT in a freshly-created
 * `git worktree`. `core.hooksPath` lives in the SHARED `.git/config` and is a
 * RELATIVE path, so every worktree resolves it against its OWN top level —
 * `<worktree>/.husky/_`, which doesn't exist without an install. Result: git
 * finds no hooks dir and runs NO hook at all, silently. The primary local
 * secret-scan guard (gitleaks, in `.husky/pre-commit`) was therefore inactive
 * in ~all worktrees, where most implementation happens.
 *
 * THE FIX: point `core.hooksPath` at `.husky` itself — a tracked directory
 * (`.husky/pre-commit` is committed) that exists in every checkout WITHOUT an
 * install. `.husky/pre-commit` is self-contained (it does not depend on the
 * `.husky/_/` runtime), so git can invoke it directly. The secret scan now
 * fires in fresh worktrees with zero `npm install` and zero auto-exec.
 *
 * SECURITY NOTE: this script introduces NO new auto-exec / supply-chain
 * surface. It performs exactly one mutation — `git config core.hooksPath
 * .husky` — plus a presence check. It runs no network calls, installs
 * nothing, and execs no arbitrary code. It does strictly LESS than the prior
 * `prepare: husky` (which spawned husky's init). We do NOT commit husky's
 * generated `_` shim (that would fight husky's design and create churn);
 * instead we make the tracked hook self-sufficient and just repoint config.
 * ---------------------------------------------------------------------------
 *
 * Idempotent. Safe to run repeatedly. Safe in CI (CI doesn't commit, so the
 * hook never fires there; this only configures local git). On a host without
 * a `.git` dir (e.g. installing a published tarball), it no-ops cleanly.
 */

import { execFileSync } from 'node:child_process';
import { existsSync } from 'node:fs';

const HOOKS_DIR = '.husky';

function main() {
  // Allow opt-out, mirroring husky's own `HUSKY=0` convention.
  if (process.env.HUSKY === '0') {
    console.log('[setup-hooks] HUSKY=0 set — skipping hook configuration.');
    return;
  }

  // No git repo (e.g. CI cache restore before checkout, or installing a
  // packed tarball outside a repo) → nothing to configure. No-op, no error.
  if (!existsSync('.git')) {
    console.log('[setup-hooks] No .git directory — skipping (not a git checkout).');
    return;
  }

  // The tracked hooks dir must exist with at least the pre-commit hook. If a
  // future refactor moves/renames it, fail loudly rather than silently
  // configuring a path with no hooks (the exact failure mode this fix closes).
  if (!existsSync(`${HOOKS_DIR}/pre-commit`)) {
    console.error(
      `[setup-hooks] ERROR: ${HOOKS_DIR}/pre-commit not found. The secret-scan ` +
        'pre-commit hook is missing from the tree. Refusing to configure an ' +
        'empty hooks path. See scaffold-source/protocols/security-scanning-defaults.md §4.'
    );
    process.exit(1);
  }

  // Point git at the TRACKED hooks dir (present in every worktree), overriding
  // husky's default `.husky/_` (untracked, absent in fresh worktrees).
  // `core.hooksPath` is shared across worktrees, so this one write covers them
  // all. execFileSync (not a shell string) → no shell-injection surface.
  try {
    execFileSync('git', ['config', 'core.hooksPath', HOOKS_DIR], { stdio: 'inherit' });
    console.log(
      `[setup-hooks] core.hooksPath = ${HOOKS_DIR} (tracked) — pre-commit secret ` +
        'scan will fire in this and every worktree.'
    );
  } catch (err) {
    console.warn(
      `[setup-hooks] WARNING: could not set core.hooksPath automatically (${err.message}). ` +
        'Run manually to restore the secret-scan guard: ' +
        `git config core.hooksPath ${HOOKS_DIR}`
    );
  }
}

main();
```

### 3. `.husky/pre-commit`

Self-contained hook: resolves the repo root, runs **gitleaks first and unconditionally** as a system binary (never gated behind `node_modules`), then runs typecheck + fast unit tests **only when `node_modules` is present**, otherwise prints a skip-notice and exits clean (so a fresh worktree's missing deps never block an otherwise-clean commit and push devs toward `--no-verify`). Make it executable: `chmod +x .husky/pre-commit`.

```bash
# Pre-commit hook — <project> (TapHQ).
#
# Fast feedback loop locally. Runs:
#   1. Secret scan (gitleaks) — primary guard per
#      scaffold-source/protocols/security-scanning-defaults.md §4 Secret
#      scanning. Catches committed credentials BEFORE they enter history
#      (where they're much harder to remove). CI re-scans as defense-in-depth.
#   2. typecheck — TS errors are usually the cheapest to fix; fail fast.
#   3. test:unit:fast — unit tests only (skips Neon-backed integration tests
#      which add ~4s; CI runs them).
#
# Total wall time on a clean tree: ~7-10 seconds.
#
# ---------------------------------------------------------------------------
# WORKTREE COVERAGE (fix/secret-scan-worktree-coverage):
#
# This hook MUST fire in fresh `git worktree` checkouts, not just the main
# clone. Most implementation here happens in worktrees (~9 live at any time).
#
# How git finds this file: `core.hooksPath` is set to `.husky` (a git-TRACKED
# directory — `.husky/pre-commit` is committed), NOT husky's default
# `.husky/_`. The `.husky/_/` shim dir is generated by `npm install`'s
# `prepare` script and is git-IGNORED (`.husky/_/.gitignore` = `*`), so it is
# ABSENT in a fresh worktree. Pointing hooksPath at `.husky/_` (husky's
# default) means git resolves a non-existent dir in fresh worktrees and runs
# NO hook at all — silently, no `--no-verify` needed. That is the gap this
# fix closes. See `scripts/setup-hooks.mjs` (the `prepare` script that pins
# hooksPath to `.husky`) and the PR for the full assessment.
#
# Because hooksPath points at `.husky`, git invokes THIS file directly. It is
# therefore fully self-contained — it does NOT source husky's `.husky/_/h`
# runtime (which is absent in fresh worktrees anyway). It also works unchanged
# if git ever invokes it via the legacy `.husky/_/pre-commit` shim (that shim
# just `sh -e`'s this same file).
# ---------------------------------------------------------------------------
#
# Required tools:
#   - gitleaks (https://github.com/gitleaks/gitleaks). Install once:
#       macOS: brew install gitleaks
#       Linux: see release page
#     If gitleaks isn't installed, this hook prints a clear error and exits
#     non-zero so the commit fails loudly — DO NOT silently skip the secret
#     scan, that defeats the protocol.
#
# OPTING OUT: `git commit --no-verify`. Use SPARINGLY — every bypass should
# have a justification. The exception is not the habit.

set -e

# Resolve the repo root so this hook behaves identically whether git invokes
# it directly (hooksPath=.husky) or via the legacy `.husky/_` shim, and
# regardless of the CWD git happens to use. `git rev-parse --show-toplevel`
# gives the top level of the worktree the commit is happening in.
ROOT="$(git rev-parse --show-toplevel)"
cd "$ROOT"

# 1. Secret scan — runs FIRST and ALWAYS, so a leaked token is caught before
# any other step has a chance to embed the credential into a test artifact or
# log. gitleaks is a system binary (brew/release), NOT a node_module, so this
# guard runs even in a fresh worktree that has never had `npm install`. This
# is the security-critical invariant: the secret scan is never gated behind
# dependency state.
if ! command -v gitleaks >/dev/null 2>&1; then
  echo "[pre-commit] ERROR: gitleaks is not installed."
  echo "[pre-commit] Required by scaffold-source/protocols/security-scanning-defaults.md §4."
  echo "[pre-commit] Install:  brew install gitleaks  (macOS)"
  echo "[pre-commit]           or see https://github.com/gitleaks/gitleaks"
  exit 1
fi
echo "[pre-commit] gitleaks (staged-only)..."
gitleaks protect --staged --redact --config=.gitleaks.toml

# 2 + 3. typecheck and unit tests need node_modules. A fresh worktree may not
# have run `npm install` (the common case — most worktrees are created with
# `git worktree add` and never installed). Running typecheck/tests there would
# hard-fail on missing deps and BLOCK an otherwise-clean commit, which would
# pressure developers toward `--no-verify` (defeating the secret scan too).
#
# So: run typecheck + tests only when node_modules is present; otherwise print
# a clear notice and skip THEM (never the secret scan). This is NOT a coverage
# loss — CI (`.github/workflows/*`) runs typecheck + the full test suite on
# every PR to dev/main. Locally, the secret scan is the primary guard; the
# typecheck/test steps are fast-feedback conveniences when deps are available.
if [ ! -x "node_modules/.bin/tsc" ]; then
  echo "[pre-commit] node_modules not installed in this worktree."
  echo "[pre-commit] Skipping typecheck + unit tests (CI runs both on every PR)."
  echo "[pre-commit] To run them locally: npm install  (then re-commit)."
  echo "[pre-commit] Secret scan PASSED — commit allowed."
  exit 0
fi

# Drop generated typedef dirs before typecheck. `.next/types/` is a Next.js
# build artifact that ALWAYS lags the active branch's routes — switching
# branches with a stale `.next/` (from a prior `next dev` or `next build`)
# surfaces phantom TS2307 errors in `.next/types/app/<old-route>/page.ts`
# files whose `src/app/<old-route>/page.tsx` no longer exists. Next.js
# regenerates these dirs on the next `npm run dev` / `next build`, so the
# rm is non-destructive. tsconfig.json explicitly includes these dirs
# (scaffold default for typed routes), so we can't fix this by tsconfig
# alone without fighting the scaffold. CI is unaffected — it builds on a
# fresh tree. See commit b1c56e6 (PR #34) for the prior bypass case.
rm -rf .next/types .next/dev/types

echo "[pre-commit] typecheck..."
npm run typecheck

echo "[pre-commit] test:unit:fast..."
npm run test:unit:fast
```

### 4. `.github/workflows/gitleaks.yml`

The authoritative CI gate. Triggers on `pull_request` AND `push` to **both** `dev` and `main` — the `pull_request: [dev]` trigger is the load-bearing one (it scans the integration-branch PR *before* merge; a `main`-only trigger would catch a leaked secret only after it already reached shared history).

```yaml
name: gitleaks

# Secret-scanning layer — implements protocols/security-scanning-defaults.md
# §A.1 row 3.
#
# CI gate (defense-in-depth): the pre-commit hook (.husky/pre-commit) is the
# intended primary local guard, but it is NOT guaranteed to have run on any
# given commit — a developer may `git commit --no-verify`, may not have
# gitleaks installed, or (historically) may be committing from a fresh
# `git worktree` where the hook silently did not fire (see
# scaffold-source/protocols/security-scanning-defaults.md §4 "Worktree
# caveat"; fixed by fix/secret-scan-worktree-coverage). Because the local
# layer cannot be assumed active on every commit, THIS CI gate is the
# authoritative secret-scan checkpoint — it must run before any merge.
#
# Triggers:
#   - pull_request → dev AND main: scan every PR BEFORE merge, on both the
#     normal feature-flow target (dev) and the prod-promotion target (main).
#     This is the pre-merge gate.
#   - push → dev AND main: scan the merged tip as a backstop (catches anything
#     that reached the branch outside the PR flow, e.g. direct/admin push).
#
# Severity: ALL findings block the PR. There's no "low-severity secret" —
# any committed credential/token/key requires rotation, full stop. The
# protocol's §4 Secret scanning section calls this out explicitly.
#
# History scan: when this workflow first activates, run `npm run leaks:full`
# locally (see package.json) once to scan the ENTIRE git history. Any
# historic secret found gets rotated even if HEAD doesn't contain it.

on:
  push:
    branches:
      - dev
      - main
  pull_request:
    branches:
      - dev
      - main

jobs:
  scan:
    name: gitleaks
    runs-on: ubuntu-latest
    steps:
      - name: Checkout (full history for diff scan)
        uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: gitleaks
        uses: gitleaks/gitleaks-action@v2
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          # Use the project's gitleaks config (.gitleaks.toml) which can
          # add project-specific allowlists. Defaults are usually sufficient.
          GITLEAKS_CONFIG: .gitleaks.toml
```

## Adapting to a new project

These snippets are copied verbatim from <project>, so they carry that project's specifics. When porting:

- **Branch names in `gitleaks.yml`:** the `dev` / `main` triggers assume the integration-branch-promotion flow. The load-bearing rule (per protocol §4) is that `pull_request` must target the project's **integration branch**, not only `main`. If a project's integration branch isn't named `dev`, change it — but never drop the integration-branch trigger down to `main`-only.
- **`scaffold-source/` path prefix in the in-code comments:** the comment lines reference `scaffold-source/protocols/security-scanning-defaults.md` because inside <project> the protocol is mirror-synced under `scaffold-source/`. In a fresh project the protocol may live elsewhere (or only here in HQ). Update the comment path to wherever the project keeps its synced copy, or drop the `scaffold-source/` prefix.
- **typecheck / test step names:** `.husky/pre-commit` calls `npm run typecheck` and `npm run test:unit:fast`. Make sure those scripts exist in the target `package.json` (or adjust the calls). The gitleaks step is the security-critical one and has no such dependency.
- **`.gitleaks.toml`:** create one at repo root (an empty file with `[allowlist]` is fine to start). Both the hook (`--config=.gitleaks.toml`) and CI (`GITLEAKS_CONFIG`) reference it.
- **`HUSKY=0` opt-out:** preserved for parity with husky's convention; `prepare` no-ops when set (useful in some CI cache-restore orderings).
