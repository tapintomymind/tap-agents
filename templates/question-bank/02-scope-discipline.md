# Question Bank — 02: Scope Discipline

**Critical dimension.** Brief items here MUST NOT be `[open]` to clear contract for `briefed` phase.

## MVP definition

- "If you had a 2-week deadline — hard, immovable — what survives in v1? Phrase as a single user flow: 'user does X, sees Y, gets Z.'"
- "Strip everything down to ONE user-visible feature. What's the feature that, if removed, kills the value? That's your MVP backbone."
- "If your MVP feature was a pull-request title, what would it be? 6-8 words."
- "Before any features — what's the user *experience* that proves the idea? A single Loom video, a manually-curated email, a Notion doc — what's the minimum that'd validate?"

## Explicit cuts

- "Name three features that are 'obviously' part of v1 that you're going to cut. If your answer is 'none of them — they're all needed,' the scope isn't tight enough yet."
- "What's the feature you'll feel embarrassed shipping without? That's the one we cut and add to v2 if needed."
- "If a competitor shipped this with half the features but better — would you ship anyway? Or would you wait to add more? Your honest answer reveals whether you're optimizing for shipping or for completeness."
- "What sounds important but isn't load-bearing for the value prop? Auth flows, settings pages, profile customization — those are common 'sounds-important' traps."

## 2-week test

- "If it has to ship in 14 days from approval, what's the cut?"
- "If it has to ship in 30 days, what gets added back? That's your phase-1.5."
- "What can you fake / Wizard-of-Oz for v1 — manual backend, hardcoded data, admin-curated content — to ship faster?"

## Must-haves vs nice-to-haves

- "List your features. Now mark each: kill / nice-to-have / must-have. Be ruthless. The 'must-have' list should be ≤4 items."
- "For each must-have: if you removed it, would users stop using the product? If 'maybe', it's a nice-to-have."

## Anti-creep

- "What's the feature you keep wanting to add even though it's not core? Architect will flag this. Let's just put it in v2 explicitly now."
- "If a friend told you 'wouldn't it be cool if it also did X', what would they say? Add X to the cut list now so we don't reinvent it during build."

## Pushback Triggers

If user gives:
- "All the features are needed" → push for explicit timeline pressure
- "It's a small project, doesn't need scoping" → push for hours/days estimate, not size
- "We can iterate" → fine, but commit to v1 scope first
- A scope that doesn't fit in 6 weeks for a solo build → push back: "Architect will say this is 12 weeks; want to revisit now?"
