# Handoff Protocol — Tier 1 → Tier 2

Defines how Tier 1 (HQ) packages work and hands it to Tier 2 (per-project execution team). Generated by Architect during the `scaffold` phase.

## Trigger

Architect produces the handoff package when:
- Phase = `scaffold` (entered after user approves planning at `planned → scaffold` checkpoint)
- `prd.md`, `scope.md`, `tech-strategy.md` all approved
- Target project repo location identified (either existing repo or new repo to be created)

## Handoff Package

Single file: `workspace/<slug>/handoff-package.md`. Self-contained — Tier 2 must be able to operate from this file alone, without needing to reach back into Tier 1's workspace mid-build.

## Tier 2 Scaffolding (writes to project repo)

After producing the handoff package, Architect writes to the target project repo:

```
<target-repo>/.claude/
├── README.md                        # Generated, includes generation date + Tier 1 reference
├── handoff-package.md               # Copy of the handoff
├── reportback.md                    # Channel back to Tier 1
├── agents/                          # Tier 2 agents (stack-specific)
│   ├── tier2-conductor.md
│   ├── <stack>-architect.md
│   ├── <component>-agent.md
│   ├── <db>-agent.md
│   └── deployment-agent.md
└── workspace/                       # Tier 2's own task tracking
    └── .gitkeep
```

The exact agent set is determined by the chosen stack and `templates/stacks/<stack>/` contents. If the stack has no template yet, Architect:
1. Notes the gap
2. Generates a baseline Tier 2 set (conductor + generic implementer)
3. Logs to `memory/agent-changelog-private.md` so Org Designer can later codify the new stack template

## Reportback Channel

`<target-repo>/.claude/reportback.md` is monitored by Tier 1's Conductor. Tier 2 appends events; Tier 1 reads.

If the target repo is in a different location than the parent `<tier1-hq>/`, Conductor needs a registered path. Stored in `workspace/<slug>/state.json`:

```json
{
  "tier2_repo_path": "<absolute path to project repo>",
  "tier2_reportback_path": "<absolute path to project repo>/.claude/reportback.md",
  "tier2_generated_at": "<timestamp>"
}
```

Example: if your Tier 1 HQ lives at `~/code/.claude/` and the project lives at `~/code/music-app/`, then `tier2_repo_path` is `/Users/<you>/code/music-app/` and `tier2_reportback_path` is `/Users/<you>/code/music-app/.claude/reportback.md`.

## Verification After Scaffold (mechanical checks)

Before transitioning `scaffold → handed-off`, Conductor verifies:
- [ ] Target `.claude/` directory exists and is non-empty
- [ ] `handoff-package.md` exists in both Tier 1 workspace AND Tier 2 location
- [ ] `reportback.md` exists at registered path (initially empty/header only)
- [ ] All agent files in Tier 2 are non-empty
- [ ] All Tier 2 agents have model assignment in frontmatter
- [ ] Tier 2 has at least: a conductor, an implementer, a critic, a deployment agent
- [ ] Tier 2 README.md generated (using `templates/tier2-readme.md`)
- [ ] A test write to Tier 2's `reportback.md` succeeds

If any check fails, scaffold is rolled back, error reported via EA.

## Critic Review of Generated Tier 2 (semantic check)

After mechanical verification passes, **Critic reviews the generated Tier 2 set** before handoff completes:

1. Conductor invokes Critic with input: the entire generated Tier 2 `.claude/` + the project's PRD + scope + tech-strategy
2. Critic asks:
   - Are all needed roles present for this PRD's scope? (e.g., payments project has security agent or deferred-with-flag)
   - Does the agent set match the chosen stack? (e.g., frontend-heavy project has component agent)
   - Are there missing capabilities the project will hit during build?
   - Do agent prompts make sense for the actual project (not just generic templates)?
3. Critic writes findings to `workspace/<slug>/critic-notes.md` with severity tags
4. **Blocking findings prevent handoff** — Architect revises Tier 2 set OR user explicitly overrides at scaffold checkpoint
5. **Warnings surface in EA's Decision Packet** at `scaffold → handed-off` for user awareness

This is the second-tier review pattern: Tier 1 Critic reviewed the *artifacts*; here Tier 1 Critic reviews the *generated Tier 2 set* before Tier 2 owns anything. After handoff, Tier 2 has its own scoped Critic for code review.

## What Tier 2 Receives vs. Does Not

**Receives:**
- Full PRD, scope, tech strategy
- Stack-specific agents
- Read access to parent memory
- Reportback channel

**Does NOT receive:**
- Tier 1's `workspace/` (other than this project's relevant artifacts as copies in handoff)
- Tier 1's other projects' state
- Direct write access to parent memory or any Tier 1 file
- Authority to advance Tier 1's state machine

## Forbidden Behaviors

- ❌ Architect scaffolding before user approves at `planned → scaffold` checkpoint
- ❌ Handoff package missing any required section
- ❌ Tier 2 created with no reportback channel established
- ❌ Tier 2 location overlapping with another project's Tier 2 (one project per repo)
- ❌ Tier 1 silently editing Tier 2's `.claude/` after handoff (changes go through promotion path; see `reportback-protocol.md`)
